# REALTEETH 채용 과제 레포지토리

본 레포지토리는 REALTEETH 프론트엔드 개발자 과제전형의 제출물 이며, 날씨 앱을 구현합니다.

## 프로젝트 실행방법

1. `pnpm i`로 의존성을 설치합니다.
2. `npx srq fix`를 실행해 `@suspensive/react-query` 관련 설정을 반영합니다
3. 필요한 환경변수를 `.env`에 설정합니다.
4. 개발 서버를 실행할 때는 `pnpm dev`를 사용합니다.
5. 프로덕션 빌드는 `pnpm build`로 생성할 수 있습니다.

빌드 시에는 빌드 스크립트 내부에서 `srq fix`가 함께 실행되므로, 별도로 한 번 더 실행하지 않아도 됩니다.

## 구현한 기능에 대한 설명

- **현재 위치 기반 날씨 조회**
  - 브라우저 Geolocation API로 현재 좌표를 가져온 뒤, 해당 위치의 현재 날씨와 당일 예보를 보여줍니다.
  - 현재 위치를 가져오지 못하거나 주소 변환에 실패한 경우에는 에러 메시지 UI를 제공합니다.
    - 에러의 원인에 따라 메세지를 다르게 표출합니다. (src/shared/lib/geolocation/error.ts)

- **지역 검색**
  - 국내 지역 카탈로그를 기준으로 지역명을 검색할 수 있습니다.
  - 검색 결과는 부분 일치 방식으로 필터링되며, 페이지네이션으로 나누어 표시합니다.

- **즐겨찾기 추가 및 관리**
  - 검색한 지역을 즐겨찾기에 추가할 수 있습니다.
  - 동일한 좌표의 지역은 중복으로 추가되지 않도록 처리했습니다.
  - 즐겨찾기는 최대 6개까지 저장할 수 있습니다.
  - 저장된 즐겨찾기는 삭제할 수 있고, 별칭을 지정하거나 수정할 수 있습니다.
  - 즐겨찾기 데이터는 persist 처리되어 새로고침 이후에도 유지됩니다.

- **즐겨찾기 상세 페이지**
  - 즐겨찾기 목록에서 특정 지역을 선택하면 상세 페이지로 이동합니다.
  - 상세 페이지에서는 해당 지역의 현재 날씨와 시간대별 날씨 정보를 더 자세히 확인할 수 있습니다.
  - 존재하지 않는 즐겨찾기 id로 접근한 경우 예외 화면을 보여줍니다.

## 사용한 기술 스택

- 프로젝트 부트스트랩, 번들링 : vite
- 외부 API 호출 : ky
- 데이터 스키마 검증 : zod
- 서버 사이드 데이터 관리 : Tanstack query (과제 필수사항)
- 라우터 라이브러리 : React router (declarative mode)
- 선언형 UI 및 api 페칭 : @suspensive/react, @suspensive/react-query
- 클라이언트 전역 상태 및 persist : zustand
- 폼 상태 관리 : react-hook-form

## 기술적 의사결정 및 이유

### 기술적 의사결정 요약 리스트

- import alias: FSD 슬라이스에 명시적 import 경로 구축
- HTTP Client: ky → fetch wrapper 대신 공통 요청 레이어 구성
- 환경변수 관리: .env → API Key 보호
- Runtime Validation: Zod → 외부 API 데이터 검증 (fail-fast)
- External API Structure: provider 기준으로 API 계층 분리
- 라우터 라이브러리: 빠르게 시작할 수 있는 React router Declarative mode
- 폼 상태 관리: 작은 입력 폼의 책임 분리를 위한 react-hook-form
- 선언형 비동기 UI: Suspense 기반 로딩/에러 처리 단순화를 위한 @suspensive/react, @suspensive/react-query
- 클라이언트 상태 관리: 즐겨찾기 공유 상태와 persist를 위한 zustand

### FSD 슬라이스를 명확히 import - import alias

FSD(Feature-Sliced Design) 구조에서는 각 레이어와 슬라이스의 경계를 명확히 유지하는 것이 중요합니다.  
하지만 상대 경로(e.g `../../shared/api/...`)를 사용할 경우, import 경로가 길어지고 현재 파일의 위치에 따라 의미가 달라져 구조 파악이 어려워집니다.

이를 해결하기 위해 Vite의 alias 기능을 활용하여 `@/`를 프로젝트 루트로 매핑하였습니다. (`vite.config.ts` 참조)

이를 통해 다음과 같은 이점을 얻을 수 있습니다:

- 슬라이스 경계를 기준으로 한 명시적인 import (`@/shared/api/...`)
- 상대 경로 깊이에 의존하지 않는 안정적인 참조
- 코드 이동 및 리팩토링 시 import 수정 비용 감소

결과적으로 import 경로 자체가 아키텍처와 레이어 규칙을 드러내도록 구성하여, 코드 가독성과 유지보수성을 개선하였습니다.

### 외부 API 호출에 관한 의사결정 - ky

외부 API 호출은 브라우저 fetch를 직접 감싸기보다, fetch 기반의 경량 HTTP 클라이언트인 ky를 사용해 공통 요청 레이어를 구성 하였습니다.

이번 과제에서는 요청 전송 자체보다, 응답 검증과 도메인 레이어 분리를 명확히 하는 것이 더 중요하다고 판단 하였습니다.

따라서 HTTP 전송에 필요한 기본 기능은 ky에 맡기고, zod 기반 검증과 에러 표준화는 별도 레이어에서 처리하도록 구성 하였습니다.

### 환경변수 관리 - vite의 .env

외부 openAPI를 호출할 때에는 api key가 필요한 경우가 많습니다.

이번 과제는 public repository에 올려지는 만큼, 불특정 다수에게 api key를 노출하기 보다, 사용 방법을 `.env.example`등으로 제시하고, 실제 `.env`를 숨겨 api key의 남용을 예방하였습니다.

### 외부 API 응답 검증 - Zod

외부 API는 클라이언트에서 제어할 수 없기 때문에 응답 데이터의 형태를 신뢰할 수 없습니다.

TypeScript는 컴파일 타임에만 타입을 보장하므로, 런타임 검증을 위해 Zod를 사용했습니다.

API 응답은 shared/api 레이어에서 검증하며, 실패 시 즉시 에러를 발생시켜 잘못된 데이터가 UI까지 전달되지 않도록 설계했습니다.

- API 스펙 변경에 대한 방어 (fail-fast)
- UI 레이어 타입 안정성 확보
- 데이터 검증 책임 분리
- 서버 사이드 데이터 관리 : Tanstack query (과제 필수사항)
- 스타일링 라이브러리 : tailwind css (과제 필수사항)

### 외부 API 계층 구조 - provider 기준 분리

외부 API 계층은 weather, geocoding 같은 도메인 기준이 아니라 OpenWeather, OpenCage 같은 provider 기준으로 분리하였습니다.

각 서비스는 base URL, API key 주입 방식, 공통 파라미터, 에러 처리 정책이 다르기 때문에 provider 단위로 클라이언트와 계약을 분리하는 편이 책임 경계를 더 명확하게 만든다고 판단하였습니다.

### 라우터 라이브러리 선택 - React Router Declarative Mode

이번 과제는 복잡한 중첩 라우팅이나 파일 기반 라우팅보다, 빠르게 화면 흐름을 구성하고 명시적으로 라우트를 관리하는 것이 더 중요하다고 판단하였습니다.

따라서 학습 비용이 낮고 React 생태계에서 널리 사용되는 React Router의 Declarative Mode를 사용하여, 필요한 라우팅 구조를 단순하게 구성하였습니다.

### 선언형 비동기 UI 구성 - @suspensive/react, @suspensive/react-query

이번 과제는 위치 검색, 날씨 조회처럼 비동기 의존성이 UI의 중심이므로, 각 컴포넌트에서 `isPending`, `error` 분기를 반복하기보다 Suspense와 Error Boundary 중심으로 화면을 구성하는 편이 더 단순하다고 판단하였습니다.

따라서 서버 상태 관리는 TanStack Query가 담당하고, 로딩/에러 표현은 `@suspensive/react`, `@suspensive/react-query`로 정리하여 비동기 UI를 보다 선언적으로 작성할 수 있도록 구성하였습니다.

### 클라이언트 상태 관리 - zustand

이번 과제에서는 서버에서 받아오는 날씨 데이터와 별개로, 사용자가 직접 관리하는 즐겨찾기 상태를 별도로 다룰 필요가 있었습니다.

즐겨찾기는 메인 페이지와 상세 페이지 양쪽에서 함께 참조되며, 장소 추가/삭제뿐만 아니라 별칭 수정 결과도 여러 화면에 즉시 반영되어야 합니다. 또한 새로고침 이후에도 상태가 유지되어야 하는 것이 UX상 유리하므로, 단순한 컴포넌트 로컬 상태나 `localStorage` 유틸만으로 관리하기보다 전역 클라이언트 상태와 persist를 함께 다루는 편이 더 적절하다고 판단하였습니다.

이에 따라 서버 상태는 TanStack Query, 사용자 로컬 상태는 zustand로 역할을 분리하여 관리합니다. 이를 통해 서버 캐시와 사용자 편집 상태의 책임을 분리하고, 즐겨찾기 관련 갱신 로직을 한 곳에서 일관되게 유지할 수 있도록 설계하였습니다.

### 폼 상태 관리 - react-hook-form

이번 과제에는 검색 입력, 별칭 수정처럼 규모는 작지만 사용자 입력을 다루는 폼 UI가 포함됩니다.

이러한 입력을 모두 `useState` 기반의 제어 컴포넌트로 처리할 수도 있지만, 폼 제출 시점의 값을 명확히 다루고 각 폼의 책임을 독립적으로 유지하기 위해 `react-hook-form`을 사용하였습니다.

특히 별칭 수정처럼 "이 지점에서 제출을 통해 값을 변경한다"는 성격의 UI에서는, 폼 상태를 컴포넌트 로컬 상태로 직접 조합하기보다 `react-hook-form`으로 초기값과 제출 흐름을 분리하는 편이 더 단순하고 읽기 쉬운 구조라고 판단하였습니다.
